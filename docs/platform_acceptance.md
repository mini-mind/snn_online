# Platform Acceptance Draft

本文档给出后续跨仓开发时的统一验收视角，防止“做完了很多代码，但无法判断是否达标”。
它是 `snn_online` 侧的协作草案，只定义希望对齐的输入输出与验收结果，不替代 `dynn` 或 `neuralsoup` 各自仓库中的实现说明。

## 1. 总体验收目标

一次最小完整链路应满足：

1. `snn_online` 能生成实验定义。
2. `dynn` 能读取该定义并执行实验。
3. `dynn` 能输出标准事件流和产物目录。
4. `neuralsoup` 能读取这些事件或产物完成可视化展示。

如果这四步不能闭环，则平台仍不算成型。

## 2. 协议层验收

### 2.1 ExperimentSpec

- 能表达任务、网络、学习规则、训练调度、运行模式。
- 不混入前端状态和底层引擎私有字段。
- 同一份 spec 能被 `snn_online` 生成、被 `dynn` 消费。

### 2.2 TopologyIR

- 能表达通用节点集合、边集合、端口和注解。
- 能表达显式边表、规则生成边、稀疏矩阵引用和双向边。
- 能通过注解和坐标系表达二维网格、层级结构等具体模板，但协议不依赖这些模板。
- 能表达塑性标签和更新权限。
- 能支持 `neuralsoup` 编辑和 `dynn` 执行。

### 2.3 RunEventSchema

- 能流式表达训练进展。
- 能表达环境状态、奖励、spike 摘要和权重更新摘要。
- 单条事件只保存轻量摘要，不直接内嵌完整 spike train、权重矩阵、环境轨迹或大体积 tensor。
- 大型数据通过产物目录和 `artifact_refs` 引用。
- 能支持实时监控和离线回放。

### 2.4 ArtifactSchema

- 目录稳定；
- 字段可版本化；
- 能支持一次 run 的复现、分析和展示。

## 3. 研究任务验收

第一阶段至少选择一个当前已有闭环任务作为统一样例：

- `point_robot`
- 或 `partial_goal_cue`

验收标准：

- 能完整执行训练和评估；
- 能记录标准指标；
- 能被前端回放展示；
- 能支持至少一组对照，例如 `LIF vs IZ`。

## 4. 性能与工程验收

### 4.1 dynn

- 在稀疏连接下运行开销应明显优于朴素稠密实现。
- 支持中断、恢复和 checkpoint。
- 有最小 profiling 指标。

### 4.2 neuralsoup

- 在中等规模拓扑下仍可流畅交互。
- 回放时不会因单次 run 数据过大而卡死。
- 渲染后端可替换，不锁死在某个实验性浏览器 API。

### 4.3 snn_online

- 实验脚本仍然清楚、简洁、可解释。
- 不因平台化而失去快速原型迭代能力。

## 5. 文档验收

在进入详细开发前，三仓至少应具备：

- 角色说明；
- 边界说明；
- 输入输出协议说明；
- 里程碑与验收标准；
- 当前实现与目标态之间的差距说明。

否则后续开发仍会反复漂移。
